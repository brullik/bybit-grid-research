# PM Acceptance

This directory contains frozen PM-owned acceptance tests and the active task scope file. Implementation PRs must not edit this directory; the PM Acceptance workflow restores it from the base branch before running acceptance checks.

Each task owns only `tasks/<task_id>/**/*.py`. A PM task-definition PR must use the same safe task ID in `active_task.json`, its task directory, and its optional frozen contract document. Task-local `conftest.py` files are forbidden.

Implementation PRs run the complete base-owned frozen suite. PM-owned PRs run the base control-plane self-tests instead; a `pm-control-plane` PR also runs a minimal staged head harness through a base-owned exact-outcome gate that requires every collected node to pass plainly. Control-plane scope excludes the active task and acceptance `conftest.py`, and protected-path validation forbids production, ordinary-test, and dependency changes. This prevents a known-invalid frozen task from circularly blocking the governance repair that is required to correct it without allowing skip/xfail padding.

## One-time frozen-test errata

A reported frozen-test defect must first be cancelled by closing the active task. After the generic erratum control plane is present on `main`, an explicitly owner-authorized `pm-frozen-erratum` PR may reactivate that same task and change exactly three paths: `active_task.json`, one existing `tasks/<task_id>/test_*.py`, and one new `errata/<task_id>.json`. The latter two files are the exact erratum payload; no contract or production file changes.

The canonical `pm_frozen_erratum_v1` manifest names an ancestor commit whose exact active-task bytes must equal the reactivated head bytes, SHA-256-pins the complete reactivated active-task bytes and the complete base and head test bytes, and records the linked issue, reason code, test path, and exact expected RED passed and failed node IDs. All complete `test_*` ASTs remain unchanged. Unsafe skip/xfail and broad-exception patterns are rejected in the mutable module/helper harness outside those immutable functions; exact runtime outcomes separately reject skips, xfails, xpasses, deselection, and collection/setup/teardown errors. The base-controlled workflow separately collects the base test and accepts the transition only when the corrected suite executes under the normal staged `RUNNER_TEMP`/`scripts` import order, covers exactly that unchanged node-ID set, and produces exactly the declared RED profile on every Python version. A successful erratum must still be followed by a fresh mandatory RED probe closed without merge, a fresh-main implementation PR, and a separate close PR.

## One-and-only second repair

When the already-merged v1 repair is itself invalid, there is no in-place edit and no reuse of `errata/<task_id>.json`. After the task is cancelled again and the owner explicitly authorizes the recovery, `pm_frozen_erratum_v2` permits exactly one second three-path PR: the reactivated `active_task.json`, the same frozen test, and a new `errata/<task_id>.v2.json`. The v1 manifest remains byte-for-byte immutable, a pre-existing v2 manifest rejects reuse, and there is no v3 path.

The compact canonical v2 manifest keeps the v1 evidence fields and adds `predecessor_commit_sha` and `predecessor_manifest_sha256`. It must match the v1 task, test, issue, historical active-task commit, head active-task hash, and v1 head-test hash. The predecessor commit must be an ancestor of the current base and contain the exact pinned v1 manifest, v1-corrected test, and reactivated active-task bytes. The checker still requires an unchanged set of `test_*` function ASTs and an exact two-list RED outcome partition.

For v2 only, baseline node IDs are collected from the current base test with trusted base dependencies so the full existing parameterization is known. The corrected head test then runs in a separately staged copy of the normal isolated harness: base `pm_acceptance`, an empty `scripts/__init__.py`, only the base-owned checker scripts, base pytest configuration, `PYTHONPATH` rooted solely at that staged `$RUNNER_TEMP`, and the installed unchanged head package. Any collection anomaly, deselection, skip, xfail/xpass, duplicate or missing call, setup/teardown failure, unexpected exit, node-set drift, or manifest outcome mismatch rejects the repair. A green v2 gate authorizes only the repaired fixture and reactivation; it resets and does not replace the mandatory closed RED probe, fresh implementation, and separate close lifecycle.

## One-time #156/#157 recovery bundle

`pm-recovery-bundle` recognizes only `p0-recovery-walk-forward-committed-key`. The activation payload is exactly the inactive-to-union `active_task.json` plus `reactivations/p0-recovery-walk-forward-committed-key.json`, introduced once by one owner-authored, non-merge commit directly on live `main`. The canonical manifest pins the historical #156 and #157 activations, active-task/test/contract hashes, exact 8+4 required paths, 32+20 node IDs, and each unavailable sentinel. The PR opens Draft; after dependency installation, Draft acceptance reconstructs the complete regular-file head tree from Git, verifies every archived byte against its commit blob ID, and rechecks every staged byte, mode, and path immediately before and after execution. The RED child starts isolated without site initialization or inherited Python/pytest paths, adds only the immutable staged source roots plus physical dependency directories, and rejects any imported `bybit_grid` or `scripts` module outside the staged tree. Thus an editable install or later checkout mutation cannot influence collection, import roots, or outcomes. All 52 declared nodes must produce one ordinary call failure and no other outcome. Only after both exact-SHA Draft matrix components pass may the owner mark the unchanged PR Ready; the `ready_for_review` run repeats the full gate and is the only merge-eligible recovery state. Every run and rerun validates the current triggering actor and positive run-attempt identity. After the ordinary implementation suite is green, close the combined task separately. A failed activation is reverted or closed; it never authorizes network, credentials, Telegram, deployment, Bybit, orders, positions, wallets, or a third erratum version.
