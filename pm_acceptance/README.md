# PM Acceptance

This directory contains frozen PM-owned acceptance tests and the active task scope file. Implementation PRs must not edit this directory; the PM Acceptance workflow restores it from the base branch before running acceptance checks.

Each task owns only `tasks/<task_id>/**/*.py`. A PM task-definition PR must use the same safe task ID in `active_task.json`, its task directory, and its optional frozen contract document. Task-local `conftest.py` files are forbidden.

Implementation PRs run the complete base-owned frozen suite in an isolated execution tree made from SHA-pinned head `src/`, `scripts/`, and ordinary `tests/` plus base-controlled `pm_acceptance`, frozen contracts, and checker scripts. The layout keeps script dependencies reachable while excluding all head acceptance controls. PM-owned PRs run the base control-plane self-tests instead; a `pm-control-plane` PR also runs a minimal staged head harness through a base-owned exact-outcome gate that requires every collected node to pass plainly. Control-plane scope excludes the active task and acceptance `conftest.py`, and protected-path validation forbids production, ordinary-test, and dependency changes. This prevents a known-invalid frozen task from circularly blocking the governance repair that is required to correct it without allowing skip/xfail padding.

## One-time frozen-test errata

A reported frozen-test defect must first be cancelled by closing the active task. After the generic erratum control plane is present on `main`, an explicitly owner-authorized `pm-frozen-erratum` PR may reactivate that same task and change exactly three paths: `active_task.json`, one existing `tasks/<task_id>/test_*.py`, and one new `errata/<task_id>.json`. The latter two files are the exact erratum payload; no contract or production file changes.

The canonical `pm_frozen_erratum_v1` manifest names an ancestor commit whose exact active-task bytes must equal the reactivated head bytes, SHA-256-pins the complete reactivated active-task bytes and the complete base and head test bytes, and records the linked issue, reason code, test path, and exact expected RED passed and failed node IDs. All complete `test_*` ASTs remain unchanged. Unsafe skip/xfail and broad-exception patterns are rejected in the mutable module/helper harness outside those immutable functions; exact runtime outcomes separately reject skips, xfails, xpasses, deselection, and collection/setup/teardown errors. The base-controlled workflow separately collects the base test and accepts the transition only when the corrected suite executes under the normal staged `RUNNER_TEMP`/`scripts` import order, covers exactly that unchanged node-ID set, and produces exactly the declared RED profile on every Python version. A successful erratum must still be followed by a fresh mandatory RED probe closed without merge, a fresh-main implementation PR, and a separate close PR.

## One-and-only second repair

When the already-merged v1 repair is itself invalid, there is no in-place edit and no reuse of `errata/<task_id>.json`. After the task is cancelled again and the owner explicitly authorizes the recovery, `pm_frozen_erratum_v2` permits exactly one second three-path PR: the reactivated `active_task.json`, the same frozen test, and a new `errata/<task_id>.v2.json`. The v1 manifest remains byte-for-byte immutable, a pre-existing v2 manifest rejects reuse, and there is no v3 path.

The compact canonical v2 manifest keeps the v1 evidence fields and adds `predecessor_commit_sha` and `predecessor_manifest_sha256`. It must match the v1 task, test, issue, historical active-task commit, head active-task hash, and v1 head-test hash. The predecessor commit must be an ancestor of the current base and contain the exact pinned v1 manifest, v1-corrected test, and reactivated active-task bytes. The checker still requires an unchanged set of `test_*` function ASTs and an exact two-list RED outcome partition.

For v2 only, baseline node IDs are collected from the current base test with trusted base dependencies so the full existing parameterization is known. The corrected head test then runs in a separately staged copy of the normal isolated harness: base `pm_acceptance`, an empty `scripts/__init__.py`, only the base-owned checker scripts, base pytest configuration, `PYTHONPATH` rooted solely at that staged `$RUNNER_TEMP`, and the installed unchanged head package. Any collection anomaly, deselection, skip, xfail/xpass, duplicate or missing call, setup/teardown failure, unexpected exit, node-set drift, or manifest outcome mismatch rejects the repair. A green v2 gate authorizes only the repaired fixture and reactivation; it resets and does not replace the mandatory closed RED probe, fresh implementation, and separate close lifecycle.

## One-time two-task recovery bundle

`pm-recovery-bundle` is a fixed owner-only transition for exactly `p0-walk-forward-exclusive-outcome-end` and `p0-committed-key-preflight`. It is selected by the sole PM mode label, requires the live sender and PR author to be `brullik`, requires canonical head/base repositories and `main`, and changes exactly `active_task.json` plus `reactivations/p0-recovery-walk-forward-committed-key.json` in one non-merge commit directly on the manifest-pinned base. A base-owned metadata-only identity preflight rejects a mismatched privileged event before checkout or head fetch, and the checker repeats that identity validation after checkout. No other task pair, manifest path, scope, or replay is supported.

The compact canonical `pm_recovery_bundle_v1` manifest pins the exact 8+4 required-path union, both historical activation commits, complete historical active-task/frozen-test/contract SHA-256 values and regular-file modes, the current task's exact one-time suspension, the previous task's single merged v1 erratum manifest and corrected test, and the base-owned exact 32+20 node IDs with their member sentinels. The checker parses and validates the canonical v1 erratum, its direct suspension parent, exact three-path diff, immutable historical digest, exact RED node partition, corrected-test bytes, and regular-file modes. It walks canonical first-parent history and rejects substituted or diverged commits, additional activations, change-then-revert state, base-side byte or mode drift, an existing manifest, stale base/head topology, or a manifest that is not new at the exact path.

The activation job collects exactly the declared 52 nodes without executing them; it cannot count as the required RED probe. A later `probe/` implementation-mode PR must produce one plain call-phase failure for every declared node, the exact member sentinel for every failure, and no pass, skip, xfail/xpass, deselection, duplicate/missing call, collection error, or setup/teardown anomaly. A non-probe implementation must instead make the complete base-owned `pm_acceptance` tree and ordinary `tests/` plain green. Ordinary tests run in a disposable pinned-head execution copy, while the immutable verifier checks the untouched source snapshots after execution; expected generated `data/processed/**` and `reports/**` files therefore remain isolated. Opening an unrelated ordinary task is blocked whenever the complete base frozen tree is not plain green, preventing a second unresolved suite from being activated.

After the combined implementation, the bundle is closed to canonical inactive in a separate task-definition PR. Every old probe or implementation remains stale. The final aggregate status also re-reads the live PR and live `main` ref and fails closed on changed head/base/main SHA or ref, labels, author, repository identity, readiness state, closed state, malformed pagination, or any unresolved review thread. An otherwise-green Draft run publishes failure, never reusable success; only a fresh unchanged-SHA Ready-triggered run with every live attestation and upstream job green may publish aggregate success.
